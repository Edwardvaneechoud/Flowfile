# Feature 9: Enhanced Code Generation

## Motivation

Flowfile's code generator converts visual flows into executable Python/Polars code. This is valuable for:

- **Auditability**: Review the exact logic in a familiar format.
- **CI/CD integration**: Deploy generated code as part of a data pipeline.
- **Version control**: Commit generated code alongside flow definitions.

However, several node types currently raise `UnsupportedNodeError`, creating gaps. Users who rely on catalog operations, database connections, or custom kernel nodes cannot generate complete code.

**Two key design principles for this feature:**

1. **Depending on `flowfile` is fine.** Generated code should `from flowfile import read_from_catalog` rather than reinventing catalog path resolution or trying to be fully standalone. The `flowfile` package is the runtime — generated code is an alternative execution path, not an escape hatch.

2. **Package output, not a single script.** Today the generator produces a flat `.py` file. With iterative nodes (embedded sub-flows), conditional execution (branch sub-graphs), and flow-as-node (referenced flows), a single script becomes unmanageable. Generated code should be a **Python package** — one module per logical unit, with clean imports between them.

## Current State

- **Code generator** (`code_generator.py`): `FlowGraphToPolarsConverter` class with 40+ node type handlers. Each handler emits Polars code for that node's operation.
- **Supported node types**: All file-based readers/writers, all transform nodes (filter, select, join, group_by, pivot, etc.), cloud storage read/write, formula, manual input, polars_code.
- **Unsupported node types** (raise `UnsupportedNodeError`):
  - `catalog_reader` — reads from catalog tables
  - `catalog_writer` — writes to catalog tables
  - `database_reader` — reads from databases (currently only PostgreSQL)
  - `database_writer` — writes to databases
  - `explore_data` — interactive exploration (output node)
  - `external_source` — external API connections
  - `python_script` — kernel-executed Python code
  - `user_defined` — custom nodes with kernel execution
- **Output format**: Single flat Python script with imports at the top, all logic inline.
- **Import management**: Accumulates imports and prepends them. Handles `polars`, `pathlib`, and cloud SDK imports.
- **Variable naming**: `node_var_mapping` (e.g., `df_1`, `df_filter_2`). Node references used when available.

## Proposed Design

### Foundational Change: Package Output

Instead of generating a single `.py` file, generate a Python package:

```
daily_sales_pipeline/
├── __init__.py
├── __main__.py              # entry point: python -m daily_sales_pipeline
├── main_flow.py             # the top-level flow logic
├── iterator_clean_data.py   # for_each sub-flow (Feature 1)
├── branch_validation.py     # condition branch logic (Feature 2)
├── sub_flow_enrichment.py   # referenced flow (Feature 8)
└── requirements.txt         # dependencies (flowfile, polars, etc.)
```

**`__main__.py`**:

```python
"""Generated by Flowfile from flow 'Daily Sales Pipeline'."""
from daily_sales_pipeline.main_flow import run

if __name__ == "__main__":
    run()
```

**`main_flow.py`**:

```python
import polars as pl
from flowfile import read_from_catalog, write_to_catalog

from daily_sales_pipeline.iterator_clean_data import process as clean_per_region
from daily_sales_pipeline.sub_flow_enrichment import process as enrich


def run():
    # Step 1: Read from catalog
    df_orders = read_from_catalog("production", "raw_orders")

    # Step 2: Clean per region (iterative node)
    df_cleaned = clean_per_region(df_orders, iterate_over="region")

    # Step 3: Enrich (sub-flow node)
    df_enriched = enrich(df_cleaned, threshold=100)

    # Step 4: Write to catalog
    write_to_catalog(df_enriched, "production", "enriched_orders")
```

**`iterator_clean_data.py`** (generated from a for_each node's embedded sub-flow):

```python
import polars as pl


def process(df: pl.DataFrame) -> pl.DataFrame:
    """Process a single partition of the for_each node."""
    df = df.filter(pl.col("status") == "active")
    df = df.with_columns(pl.col("amount").round(2).alias("amount_rounded"))
    return df
```

**`sub_flow_enrichment.py`** (generated from a referenced flow):

```python
import polars as pl
from flowfile import read_from_catalog


def process(df: pl.DataFrame, threshold: float = 50) -> pl.DataFrame:
    """Generated from flow 'Data Enrichment Pipeline' (catalog ID: 42)."""
    ref_data = read_from_catalog("production", "reference_table")
    df = df.join(ref_data, on="id", how="left")
    df = df.filter(pl.col("score") > threshold)
    return df
```

### Phase 1: Catalog Read/Write

Use `flowfile` imports — no need to reinvent catalog path resolution:

```python
from flowfile import read_from_catalog, write_to_catalog

df = read_from_catalog("namespace", "table_name")
# ... transformations ...
write_to_catalog(df, "namespace", "output_table")
```

If time travel (Feature 3) is available:

```python
df = read_from_catalog("namespace", "table_name", version=5)
```

### Phase 2: Database Read/Write

Use `flowfile` database helpers wrapping the existing PostgreSQL implementation:

```python
from flowfile import read_from_database, write_to_database

df = read_from_database(
    connection_name="orders_db",      # references a registered connection
    query="SELECT * FROM orders WHERE date > '2024-01-01'",
)

write_to_database(df, connection_name="warehouse", table_name="output")
```

Credentials are resolved through Flowfile's connection registry and secrets system — never hardcoded in generated code.

### Phase 3: Kernel Code (Custom Nodes / Python Script)

Custom nodes and python_script nodes contain user-written Polars code. Each gets its own module in the package:

**`python_script` node** → separate function:

```python
# custom_score_3.py
import numpy as np
import polars as pl


def process(df: pl.DataFrame) -> pl.DataFrame:
    """Python script node (node 3)."""
    return df.with_columns(pl.Series("score", np.random.rand(len(df))))
```

**`user_defined` (custom) node** → extract the `process()` method:

```python
# custom_node_validator_4.py
import polars as pl


def process(df: pl.DataFrame, threshold: float = 100) -> pl.DataFrame:
    """Custom node 'Validator' (node 4)."""
    return df.filter(pl.col("value") > threshold)
```

The main flow module imports and calls these:

```python
from daily_sales_pipeline.custom_score_3 import process as compute_score
from daily_sales_pipeline.custom_node_validator_4 import process as validate

df = compute_score(df)
df = validate(df, threshold=200)
```

### Phase 4: Iterative and Conditional Code

**For-each nodes** (Feature 1): The embedded sub-flow becomes a module with a `process()` function. The main flow calls it via a partition-and-apply pattern:

```python
from daily_sales_pipeline.iterator_clean_data import process as clean_partition

# Scatter-gather
results = []
for region, partition_df in df.partition_by("region", as_dict=True).items():
    results.append(clean_partition(partition_df))
df_cleaned = pl.concat(results)
```

**Condition nodes** (Feature 2): Each branch becomes a function:

```python
from daily_sales_pipeline.branch_active import process as handle_active
from daily_sales_pipeline.branch_inactive import process as handle_inactive

df_active = df.filter(pl.col("status") == "active")
df_inactive = df.filter(pl.col("status") != "active")

df_active = handle_active(df_active)
df_inactive = handle_inactive(df_inactive)
df = pl.concat([df_active, df_inactive])
```

### Phase 5: Import Management

Generated code should follow Python conventions:

```python
# stdlib
from pathlib import Path

# third-party
import polars as pl

# flowfile
from flowfile import read_from_catalog, write_to_catalog

# local package
from daily_sales_pipeline.iterator_clean import process as clean_partition
```

Rules:
1. Deduplicate across all modules.
2. Only import what's used.
3. Group: stdlib → third-party → flowfile → local.
4. `requirements.txt` lists all needed packages.

## Open Questions

1. **Package naming**: Should the package name be derived from the flow name, or user-specified? Sanitization rules for Python package names.
2. **`explore_data` nodes**: Interactive — no code can be generated. Silently skip, or emit a `# TODO: explore_data node skipped` comment?
3. **External sources**: API-based external sources require live connections. Emit a `# TODO` placeholder, or generate `requests` code?
4. **Backward compatibility**: Existing users expect a single `.py` file. Should we keep single-file mode as an option for simple flows (no sub-flows, no iterators)?
5. **Round-trip fidelity**: How close should the generated code's output match the flow's output? Should this be tested?
6. **`flowfile` public API**: The convenience functions (`read_from_catalog`, etc.) need to be part of a stable public API. What's the right module structure?
